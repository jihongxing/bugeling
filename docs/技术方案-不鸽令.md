## 3. 核心云函数设计

### 3.1 活动管理

| 云函数 | 触发方式 | 功能 |
|--------|----------|------|
| `createActivity` | 用户调用 | 创建活动，校验参数，调用 msgSecCheck 审核标题和接头特征 |
| `getActivityList` | 用户调用 | 基于 LBS 查询 20km 内活动，按距离排序 |
| `getActivityDetail` | 用户调用 | 获取活动详情，根据时间条件决定是否返回发起人微信号 |
| `approveParticipant` | 发起人调用 | 同意参与者加入，更新状态为 `approved`，生成核销码 |

### 3.2 支付与资金

| 云函数 | 触发方式 | 功能 |
|--------|----------|------|
| `createDeposit` | 用户调用 | 调用微信支付统一下单，创建押金支付订单 |
| `payCallback` | 微信回调 | 支付成功回调，更新参与记录状态为 `paid` |
| `refundDeposit` | 系统调用 | 核销成功后触发，调用微信退款接口全额退款 |
| `splitDeposit` | 系统调用 | 违约后触发，调用微信分账接口按 3:7 分账 |

### 3.3 核销逻辑

| 云函数 | 触发方式 | 功能 |
|--------|----------|------|
| `generateQrToken` | 用户调用 | 生成动态核销码 token，有效期 60 秒，过期自动刷新 |
| `verifyQrToken` | 发起人扫码 | 校验 token 有效性 + 活动匹配 + 参与者身份，成功则触发退款 |
| `reportArrival` | 用户调用 | 记录"我已到达"的时间和 LBS 坐标 |

### 3.4 自动仲裁

| 云函数 | 触发方式 | 功能 |
|--------|----------|------|
| `autoArbitrate` | 定时触发器（每分钟） | 扫描所有 `confirmed` 状态且已超时 60 分钟的活动，执行仲裁逻辑 |

仲裁决策树：

```
活动超时 60 分钟且未核销
│
├─ 参与人位置偏差 > 1000m？
│  └─ YES → 参与人违约，3/7 分账，参与人信用 -20
│
├─ 发起人位置偏差 > 1000m？
│  └─ YES → 发起人违约，全额退款，发起人信用 -20
│
├─ 双方均在场但未核销？
│  └─ YES → 视为参与方放弃，3/7 分账补偿发起人
│
└─ 双方均未到场？
   └─ YES → 互鸽，原路退款，双方信用 -5
```

### 3.5 信用分管理

| 云函数 | 触发方式 | 功能 |
|--------|----------|------|
| `updateCredit` | 系统内部调用 | 根据核销/违约/举报事件更新信用分 |
| `checkCreditAccess` | 用户操作前 | 检查用户信用等级，< 80 限制发起，< 60 禁止使用 |

### 3.6 内容安全

| 云函数 | 触发方式 | 功能 |
|--------|----------|------|
| `checkTextSafety` | 发布时调用 | 调用微信 `msgSecCheck` 检查标题和接头特征 |
| `checkImageSafety` | 上传时调用 | 调用微信图片安全接口检查上传图片 |

### 3.7 活动日历与信用海报

| 云函数 | 触发方式 | 功能 |
|--------|----------|------|
| `getCalendarActivities` | 用户调用 | 查询用户月度活动（发起人+参与者），按日期分组，计算状态颜色和月度统计（守约率、补偿金额、计划支出） |
| `checkConflict` | 报名前调用 | 检测新活动与已有活动的时间重叠和路程风险（Haversine 距离 > 5km 且间隔 < 60min） |
| `getPosterData` | 用户调用 | 计算海报数据：日历颜色点、信用分、击败百分比、文案生成 |

冲突检测集成在活动详情页的报名流程中，调用失败不阻塞报名。

## 4. 状态机详细设计

### 4.1 活动状态

```
pending（待组队）
  │
  ├─ 有参与者被同意 → confirmed（已成行）
  │                      │
  │                      ├─ 扫码核销成功 → verified（已核销）
  │                      │                    └─ 触发全额退款
  │                      │
  │                      └─ 超时 60 分钟 → expired（已超时）
  │                                           └─ 进入自动仲裁
  │                                                └─ settled（已结算）
  │
  └─ 超过活动时间无人参与 → cancelled（已取消）
```

### 4.2 参与记录状态

```
paid（已支付）
  │
  ├─ 发起人同意 → approved（已通过）
  │                  │
  │                  ├─ 扫码核销 → verified（已核销） → refunded（已退款）
  │                  │
  │                  └─ 超时仲裁 → breached（违约） → settled（已结算）
  │
  └─ 发起人拒绝 → rejected（已拒绝） → refunded（已退款）
```

## 5. 核销码安全方案

### 5.1 动态 Token 机制

- 核销码本质是一个带签名的 JWT token，包含：`{ activityId, participantId, timestamp, nonce }`
- 有效期 60 秒，前端每 50 秒自动请求刷新
- 二维码内容 = token 字符串，发起人扫码后由云函数校验

### 5.2 防作弊措施

- token 一次性使用，核销后立即失效
- 服务端校验 token 签发时间，超过 60 秒拒绝
- 同一 participation 同时只能有一个有效 token

## 6. 微信支付集成

### 6.1 支付流程

```
参与者点击报名
  → 前端调用 createDeposit 云函数
    → 云函数调用微信统一下单 API
      → 返回支付参数给前端
        → 前端调起微信支付
          → 支付成功 → 微信回调 payCallback
            → 更新 participation 状态为 paid
```

### 6.2 退款流程（履约成功）

```
发起人扫码核销成功
  → verifyQrToken 云函数验证通过
    → 调用微信退款 API（全额原路退回）
      → 更新 participation 状态为 refunded
        → 更新双方信用分 +2
```

### 6.3 分账流程（违约）

```
autoArbitrate 判定违约
  → 24 小时申诉缓冲期
    → 无申诉 → 调用微信分账 API
      → 30% 划入平台账户
      → 70% 划入发起人账户
        → 更新 participation 状态为 settled
          → 更新违约方信用分
```

### 6.4 关键注意事项

- 所有金额以"分"为单位存储和计算，避免浮点精度问题
- 退款和分账均走微信原路，不进用户余额，规避二清
- 分账需提前在微信商户平台配置分账接收方

## 7. LBS 地理服务

### 7.1 活动列表查询

- 使用云数据库 GEO 地理位置查询，以用户坐标为圆心，20km 为半径
- 数据库 location 字段建立地理位置索引

### 7.2 到场校验

- 用户点击"我已到达"时，前端调用 `wx.getLocation` 获取高精度坐标
- 云函数计算用户坐标与活动地点的距离（Haversine 公式）
- 偏差阈值：1000 米

### 7.3 腾讯地图 POI

- 发布活动时调用腾讯地图 POI 搜索，确保选择的是公共场所
- 存储 POI 的 name、address、latitude、longitude

|------|------|------|

| 过期活动清理 | 每天 | 清理超过 7 天的已完结活动数据 |
