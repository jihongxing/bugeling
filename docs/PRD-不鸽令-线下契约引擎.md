# 产品需求文档 (PRD) - 不鸽令·线下契约引擎

**产品版本：** v1.2  
**产品定位：** 极简线下约见履约工具  
**核心价值：** 消除线下约见的不确定性，为"时间"和"信用"定价  
**核心机制：** 预付"鸽子费"（押金） → 线下扫码核销 → 自动退款/违约分账

---

## 1. 核心机制设计

### 1.1 鸽子费（押金）梯度

- **金额设定：** ¥9.9、¥19.9、¥29.9、¥39.9、¥49.9（由发起人发布时选定）。
- **目的：** 标准化金额降低用户纠结，同时覆盖不同价值档位的活动（如奶茶约见 vs 晚饭约见）。

### 1.2 资金流转（3/7 分账）

- **履约成功：** 发起人扫码确认后，押金 100% 即时原路退回参与人。
- **违约扣款：**
  - 平台收益 (30%)：用于覆盖微信支付手续费、系统运营及研发成本。
  - 发起人补偿 (70%)：作为对发起人时间损失和信誉损失的直接经济补偿。

### 1.3 社交解锁：微信联系逻辑

- **隐私保护：** 活动组队成功前，双方不可见联系方式。
- **条件解锁：** 活动开始前 2 小时，参与者页面自动显现"复制发起人微信"按钮。
- **目的：** 保证最后的即时通讯，又防止过早脱离平台导致契约失效。

---

## 2. 用户路径 (User Journey)

### 2.1 发起方 (Initiator)

1. **发布：** 选定 POI（公共地点）、时间、人数、鸽子费档位、接头特征（必填）。
2. **筛选：** 查看报名列表（仅限已支付押金的用户），点击"同意"即组队成功。
3. **核销：** 线下见面，打开小程序扫描参与者活动码。
4. **评价：** 简单标记对方是否守时。

### 2.2 参与方 (Participant)

1. **查找：** 浏览附近活动。
2. **支付：** 选定活动后，通过微信支付预缴鸽子费。
3. **凭证：** 组队成功后，系统自动生成唯一的"活动核销码"。
4. **解锁：** 活动开始前 2 小时，解锁"复制发起人微信"功能，进行最后沟通。
5. **见面：** 向发起人出示核销码，扫码后收到微信支付退款通知。

---

## 3. 功能模块需求

### 3.1 首页 & 活动列表

- **LBS 聚合：** 以用户当前位置为圆心，展示 20km 内的契约请求。
- **硬核标签：** 突出显示"鸽子费金额"及"当前人数"。

### 3.2 发布模块（极简表单）

- **主题：** 手动输入（如"三和路口200元吃喝一天"）。
- **人数：** 1-20 人。
- **鸽子费选择：** 单选框（9.9 / 19.9 / 29.9 / 39.9 / 49.9）。
- **地点：** 调用腾讯地图 POI 选择，确保是公共场所。
- **接头特征：** 必填项，用于线下辨认。

### 3.3 核销与状态机

- **状态流转：**
  - `待组队` → `已成行（已交费）` → `已核销（退款）`
  - `已成行` → `超时未核销` → `结算分账`
- **核销码安全性：** 二维码需具备动态刷新机制，防止截图作弊。
- **核销流程：**
  1. 发起人点击"生成核销码"。
  2. 参与人出示二维码，发起人扫码。
  3. 触发动作：微信支付后台立即发起退款申请，原路退回押金。

---

## 4. 信用与自动仲裁系统

### 4.1 契约分 (Credit Score)

- **初始分：** 100 分。
- **加分：** 每次扫码核销成功 +2 分。
- **扣分：**
  - 系统判定放鸽子：-20 分。
  - 被举报且核实：-30 分。
- **等级权益：**
  - 信用分低于 80 分：限制发起活动次数。
  - 信用分低于 60 分：禁止使用平台。

### 4.2 自动仲裁机制 (Auto-Arbitration)

针对"未核销"状态，系统在活动开始 60 分钟后自动执行以下逻辑：

1. **位置校验（LBS）：** 系统调取双方最后一次点击"我已到达"时的经纬度。
2. **裁决结果：**
   - **参与人未到场**（位置偏差 > 1000米）：判定参与人违约，执行 3/7 分账。
   - **发起人未到场**（位置偏差 > 1000米）：判定发起人违约，全额退还参与人押金，并重扣发起人信用分。
   - **双方均到场但未核销：** 视为"见后变卦"，根据契约铁律，未进行核销即视为参与方放弃参与，执行 3/7 分账补偿发起人。
   - **双方均未到场：** 视为"互鸽"，押金原路退回，但双方均扣除 5 分信用分。

## 5. 关键界面文案（契约声明）

在参与人支付押金页面及活动详情页，必须以高亮醒目方式展示以下声明：

> **【契约声明】**
>
> 本平台鸽子费旨在保障双方的时间成本。一旦您支付押金并成功组队，即视为接受本契约：
>
> 1. 您必须在约定时间内到达约定地点并完成核销。
> 2. 除非对方存在违法或严重违规行为，否则任何基于"眼缘不合"、"临时变卦"、"主观不想参与"的原因未完成核销的，均视为违约。
> 3. 违约情况下，押金将按比例扣除并补偿给发起人。
>
> 请在充分确认对方信息后再行报名。

## 6. 异常处理与投诉

- **发起人放鸽子：**
  - 参与者可在现场点击"发起人未到场"并拍照存证（带 LBS 地理信息）。
  - 核实后，押金退还参与者，并对发起人进行封号处理。
- **人货不符举报：** 参与者到达现场发现发起人与描述严重不符（如非法交易暗示），可点击"紧急举报并冻结"。
- **人工介入：** 仅在双方均在现场且均提起投诉的极端情况下，进入人工后台核实照片存证。
- **退款路径：** 统一走微信原路退款（不进用户个人余额，确保合规）。

## 7. 产品合规与安全防护

### 7.1 去社交化

- 小程序内不设对话框，只提供"确认按钮"。
- 如需沟通，引导用户使用微信原生社交功能（点击复制对方微信号）。

### 7.2 内容风控

- 接入微信 `msgSecCheck` 接口。
- **拦截词库：** 包含但不限于"上门、私密、特殊服务、药、暗语"等。
- **图片检测：** 严禁发布色情、暴露或带有联系方式的图片。
- 对发起人上传的"接头特征"进行 AI 文本/图片审核，严禁出现色情暗语。

### 7.3 资金结算合规

- 使用微信支付"分账"接口，采用延迟结算模式。
- 核销成功：立即退款。
- 超时未核销：24 小时后执行分账，确保有足够的申诉缓冲期。
- 每笔违约金系统自动根据 3:7 比例划拨，不留存资金池，规避二清风险。

## 8. 技术架构与开发路线

### 8.1 推荐架构

微信小程序 + 云开发 (CloudBase) + 云数据库

### 8.2 开发路线图

1. **Phase 1：** 微信小程序基础架构 + LBS 地理围栏校验。
2. **Phase 2：** 微信支付分账接口集成 + 2 小时自动解锁逻辑。
3. **Phase 3：** 自动仲裁算法引擎及信用分数据库建设。

### 8.3 开发周期预估

| 模块 | 工期 |
|------|------|
| UI 设计 | 2 天 |
| 前端开发 | 4 天 |

| **合计** | **约 11 个工作日** |

## 9. 活动日历与信用海报

### 9.1 活动日历（我的契约日程）

位于"我的"页面显著位置，以日历视图按日期展示用户参与的所有活动。

**核心定位：** 从"日程表"到"信用足迹"——不是冷冰冰的记事本，而是契约仪表盘。

**状态颜色标识：**
- 绿色圆点：已完结（成功核销，押金已退）
- 黄色圆点：待进行（已付押金，尚未到期）
- 红色圆点：违约（因放鸽子被扣款）
- 灰色圆点：已失效（未成行）

**防冲突与缓冲区：**
- 防撞车警告：用户尝试加入时间重叠的活动时，弹出提示"契约冲突！您在那段时间已有一场不鸽令，强行加入若无法准时到达，将损失两份押金。"
- 路程缓冲区：根据两个活动的 LBS 位置自动计算。若两个活动地点相隔较远但时间间隔过短，标红预警"行程过紧，鸽子风险极高！"

**日历功能：**
- 月视图：日期下方带彩色状态点，一目了然
- 今日任务置顶：显示今天的集合地点、接头特征和"复制微信"按钮
- 信用总结：日历上方显示"本月守约率 XX%，累计节省/获赠补偿 ¥XX"
- 预算汇总：日历底部显示"本月已计划支出"（累计鸽子费 + 预计消费）
- 历史回顾：点击历史日期查看活动详情（和谁见面、接头特征、互评标签）

### 9.2 信用海报（分享守约月报）

用户可将月度日历生成一张海报分享到微信群或朋友圈。

**海报内容：**
- 月度日历缩略图（满屏绿色圆点 = 信用极好）
- 守约统计：本月守约 X 次，从未放鸽子
- 契约分展示
- 推广文案示例："三月份，我在线下守约 12 次，从未放鸽子。这就是我的契约精神。"

**目的：** "秀信用"行为成为产品口碑传播的天然载体。
